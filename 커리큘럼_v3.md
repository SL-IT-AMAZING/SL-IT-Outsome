## Week 1. 설계주 — MVP 설계 + 미국 시장 이해

> "이런 거 만들고 싶어"라는 막연한 아이디어가 대화 3번이면 설계서 3개로 바뀝니다

### 🔨 BUILD — 바이브코딩

**프롬프트 기초 (6클립)**

- 1-1. ⭐ 5분 만에 랜딩페이지 완성 — 이론은 나중에, 결과부터
- 1-2. 같은 AI한테 시켰는데 왜 결과가 다를까 — 프롬프트 before/after
- 1-3. 프리뷰에서 클릭하고 "이거 바꿔줘" — 제목은 크게, 버튼은 눈에 띄게
- 1-4. GitHub, Git 몰라도 됩니다 — 체크포인트 한 번이면 언제든 되돌리기
- 1-5. "Notion 랜딩페이지처럼 해줘" — 10분 만에 카피하는 법
- 1-6. 여기서 99%가 막힙니다 — Part 1 에러 TOP 5

**Alpha·Beta·Gamma 설계 (10클립)**

- 1-7. 코드 치기 전에 설계부터 — 대화 3번이면 설계서 3개, 개발은 AI가
- 1-8. ⭐ Alpha: "이런 스타트업 만들고 싶어" — AI가 질문하고, 당신이 답하면 PRD가 완성됩니다 `💡 G0-1, G1-2 먼저 수강 권장`
- 1-9. Alpha 실전: 막연한 아이디어를 구체적 기획서로 — 좋은 대화 vs 나쁜 대화
- 1-10. Beta: "사용자가 어떤 순서로 쓸까?" — AI와 대화하며 유저플로우 그리기 `💡 G1-2 먼저 수강 권장`
- 1-11. Beta 실전: "이 흐름에서 빠진 게 있지 않아?" — AI가 놓친 부분 잡아내는 법
- 1-12. Gamma: "데이터를 어떻게 저장하지?" — AI와 대화하며 ERD 설계하기
- 1-13. Gamma 실전: "이 데이터, 유저한테 붙여? 설문에 붙여?" — 모호한 관계를 대화로 정리하기
- 1-14. 설계서 3개 리뷰 — 여기서 빠지면 개발 중에 3배 고생합니다
- 1-15. 🎯 [실습] 내 MVP 설계 완성: 아이디어 → Alpha·Beta·Gamma 대화 → 설계서 3개
- 1-16. 설계를 대충 하면 벌어지는 일 — Part 2 에러 TOP 5

### 🌏 GROW — Outsome

- G1-1. 미국 시장 이해 — 한국 벤처와 무엇이 다른가 (멘토: Apoorva — AI & Agent, 현지 VC 관점)
- G1-2. 스타트업 가설 수립 + 미국 BM 초안 — "누구의 어떤 문제를 푸는가?" (Peter Shin 1:1 Office Hour) → 산출물: **가설 캔버스** + **BM 초안**

> **🔗 BUILD × GROW 시너지**: GROW에서 작성한 **가설 캔버스**(누구+환경+문제+현재비용+솔루션)가 → BUILD Alpha 대화에 직접 입력됩니다. "이런 스타트업 만들고 싶어"가 아니라, **가설 캔버스 기반**으로 AI와 대화하니 PRD 품질이 완전히 달라집니다. GROW에서 작성한 **BM 초안**이 → Beta 유저플로우의 결제/과금 흐름 설계에 반영됩니다.

## Week 2. 빌딩주 — MVP 빌딩 + 고객 검증

> 설계서 넣고 "만들어줘" 했더니 진짜 나왔습니다 — 이제 다듬을 차례입니다

### 🔨 BUILD — 바이브코딩

- 2-1. 설계서 넣고 "만들어줘" 한 마디 — 5분 후 코드가 쏟아집니다 `💡 G2-1, G2-2 먼저 수강 권장`
- 2-2. ⭐ AI가 만든 내 MVP, 처음 열었는데 진짜 돌아간다
- 2-3. "이건 맞고, 이건 고쳐줘" — 결과물 리뷰하는 법
- 2-4. "여기 간격 좁혀줘, 이 색 바꿔줘" — 프리뷰에서 클릭하고 수정
- 2-5. UI가 구린 건 프롬프트 탓 — "예쁘게 해줘" 말고 이렇게
- 2-6. 에러 메시지 복붙하면 AI가 고쳐줍니다
- 2-7. "이 기능도 추가해줘" — 설계서에 없던 기능, 한 마디면 됩니다
- 2-8. 🎯 처음부터 끝까지 직접 눌러보세요 — 내 MVP 전체 점검
- 2-9. 빌드 에러, 토큰 초과 — Part 3 에러 TOP 5

### 🌏 GROW — Outsome

- G2-1. GTM 설계: 고객 인터뷰 + 프리토타이핑 — MVP 만들기 전에 "진짜 원하는지" 검증하는 법 (멘토: Keeyong — 파운더 마인드셋) → 산출물: **인터뷰 인사이트 시트**
- G2-2. ICP 확정 + GTM 실행 리뷰 — 데이터로 고객을 증명합니다 (Peter Shin 1:1 Office Hour)

> **🔗 BUILD × GROW 시너지**: GROW에서 작성한 **인터뷰 인사이트 시트**(고객 인터뷰 결과 + 프리토타이핑 데이터)가 → BUILD MVP의 가설을 고도화합니다. "이 문제가 진짜 존재하는가?"를 검증한 뒤, **고객이 실제로 겪는 문제**에 집중해서 만듭니다. 프리토타이핑에서 반응 좋았던 가설 → MVP 핵심 방향으로 반영. 반응 없던 가설 → 과감히 피봇. **피봇이 두렵지 않습니다 — Anyon이면 새 아이디어도 24시간 안에 다시 MVP로 만들 수 있습니다.** 실제로 인터뷰 과정에서 아이템이 완전히 바뀌는 팀도 많습니다. 배포 후에는 실제 사용자 피드백이 제품 개선의 입력값이 됩니다.

## Week 3. 런칭주 — 배포 + KPI/Pricing

> 백엔드 연동, 배포 원클릭 — 내 스타트업에 전 세계가 접속합니다

### 🔨 BUILD — 바이브코딩

- 3-1. 백엔드 연동 — 클릭 한 번이면 데이터 저장 끝 (Supabase 실습) `💡 G3-1 먼저 수강 권장`
- 3-2. 로그인/회원가입 — 구글 로그인까지, 전체 과정에서 딱 1번 직접 하는 순간
- 3-3. ⭐ 배포 — 원클릭으로 세상에 공개 (Vercel 실습)
- 3-4. myservice.com — 내 도메인 연결하는 법
- 3-5. 배포 후 "안 돼요" 연락 왔을 때 — 수정하고 다시 올리는 법
- 3-6. 🎉 첫 고객을 만날 준비 완료 — 당신의 MVP가 살아있습니다

### 🌏 GROW — Outsome

- G3-1. KPI & 프라이싱 — 실리콘밸리 표준으로 숫자 세우기 + AARRR 퍼널 설계 (멘토: Jina — US CS) → 산출물: **KPI 대시보드** + **프라이싱 모델**
- G3-2. 폭리를 취하는 법 — 가격을 못 정하는 진짜 이유와 Day-1 수익 구조 (Peter Shin 1:1 Office Hour)

> **🔗 BUILD × GROW 시너지**: GROW에서 설계한 **KPI 대시보드 + 프라이싱 모델**(AARRR 퍼널 기반)이 → BUILD 배포에 직접 반영됩니다. 배포할 때 트래킹 코드를 심고, **KPI 대시보드**에서 정한 지표를 실시간 측정합니다. **프라이싱 모델**에서 검증한 가격으로 만듭니다. "배포하고 끝"이 아니라 **"배포하고 측정"**입니다.
